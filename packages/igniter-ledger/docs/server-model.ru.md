# Contract-Native Store: Серверная модель и транспортная архитектура

Дата статуса: 2026-04-30.
Аудитория: Architect Supervisor, Package Agent, Research.

---

## Контекст

`igniter-ledger` начинался как встроенный POC — приложение и хранилище в одном
процессе. Это работает для демо и однонодовых деплоев, но создаёт жёсткое
ограничение: хранилище живёт и умирает вместе с процессом приложения.

Архитектурный вопрос, который мы исследовали: **как хранилище должно расти,
когда приложение масштабируется за пределы одного процесса, или когда источники
данных внешние (IoT датчики, event streams, удалённые сервисы)?**

---

## Модель Проекции

Ключевая идея: приложение не встраивает хранилище. Вместо этого оно **деплоит
свой контракт на сервер**, который становится хостом проекции.

```
Процесс приложения          Сервер хранилища
──────────────────          ──────────────────────────
Определение контракта  ───► Contract Registry
                             │
Вычисления в рантайме        │  (хранилища держат иммутабельные факты,
(остаются в приложении)      │   отдают scopes + replay)
                             │
                     ◄────── Чтение фактов (scope/replay/causation)
```

Ключевые свойства:
- **Вычисления остаются в приложении.** Сервер не выполняет узлы контракта.
- **Сервер — это долговечное хранилище фактов.** Он хранит WAL, управляет
  чекпоинтами, обслуживает запросы.
- **Интерфейс — данные, не коллбэки.** Приложение отправляет вызовы
  `write`/`append`; сервер отвечает фактами. Никакого RPC для логики.

## Модель Доставки Событий

Сервер пишет события — он не решает, кто их получит и как.
Доставка — **ответственность пользователя**, настраивается через DSL-адаптеры:

```ruby
# Объявление подписки на стороне приложения
store.on_scope(Task, :open) do |adapter(:webhook, url: "https://myapp.com/tasks")|
  # адаптер обрабатывает доставку; сервер стреляет событием → адаптер берёт управление
end

store.on_scope(Task, :open, adapter: :queue, queue: "task-changes")
store.on_scope(SensorReading, :recent, adapter: :sse, channel: "sensors")
```

**Поток через Event Bus:**

```
Сервер: факт записан
   └─► SubscriptionRegistry — находит подходящие SubscriptionRecord
         └─► Event Bus
               ├─► WebhookAdapter    (HTTP POST, retry)
               ├─► QueueAdapter      (SQS / Sidekiq / Kafka)
               ├─► SSEAdapter        (Server-Sent Events / WebSocket)
               └─► (пользовательский адаптер)
```

Сервер пишет в свой лог событий и возвращает управление. Доставка — async и
подключаемая. Вызов `on_scope` становится постоянным `SubscriptionRecord`
(не Ruby Proc), когда используется сетевой бэкенд.

## Топологии Деплоя

Ключевое понимание: это не разные продукты — это разные **конфигурации бэкенда**
одного и того же фасада `Companion::Store`. Код приложения не меняется.

### 1. Встроенный (текущий дефолт)

```
[App + IgniterStore]
```

- Бэкенд: `:memory` или `:file`
- Один процесс; нулевые сетевые издержки
- Работает сейчас; хорошо для демо, однонодового деплоя, разработки

### 2. Локальный сервер (IPC)

```
[Процесс App] ──unix socket──► [Процесс IgniterStore Server]
```

- Бэкенд: `:network, transport: :unix`
- Одна машина; субмиллисекундная задержка
- Падение приложения не теряет состояние хранилища

### 3. Общий Кластер

```
[App 1] ──► [Store Cluster (primary + replicas)]
[App 2] ──►      │
[App N] ──►      └─► WAL реплицируется через replication log
```

- Бэкенд: `:network, transport: :tcp, address: "store.internal:7400"`
- Несколько инстансов приложения используют одну проекцию
- Строгая консистентность через кворумную запись (для `:strong` сторов)

### 4. Edge / IoT

```
[Сенсорный узел]
   └─► локальный IgniterStore (принимает немедленно, `:eventual`)
         └─► async репликация ──► Центральный Store
```

- Локальный стор принимает записи без координации
- Репликация — фоновая и бесконфликтная (семантика grow-only set)
- Центральный стор объединяет через union (CRDT)

---

## CRDT-Совместимые Факты

Факты Igniter **иммутабельны и content-addressed**:
- Каждый факт имеет UUID `id` и `value_hash` (BLAKE3 стабильно-сортированного JSON)
- Факты никогда не обновляются — новая запись создаёт новый факт с новым id
- FactLog — это **grow-only set**: `merge(A, B) = A ∪ B`

Это означает:
- Никаких конфликтов записи (два узла могут оба аппендить; union всегда консистентен)
- Gossip-репликация безопасна: обмениваемся наборами фактов, объединяем через union
- Raft не нужен для eventually-consistent сторов
- Time-travel и causation chain сохраняются через слияния

**Causation chain** (`fact.causation = prior_fact.id`) выживает при репликации
неизменным — он content-addressed и иммутабельный.

## Аннотация Консистентности

Не все сторы требуют одинаковой гарантии консистентности.
Аннотация на уровне схемы в момент определения:

```ruby
# Strong: кворумная запись до подтверждения (финансовые записи, audit log)
store.register(Transaction, consistency: :strong)

# Eventual: локальное принятие + async репликация (показания датчиков, телеметрия)
store.register(SensorReading, consistency: :eventual)
```

Консистентность — это **свойство схемы**, не соединения.
Сетевой бэкенд маршрутизирует записи соответственно:
- `:strong` → ждёт кворумного подтверждения перед возвратом
- `:eventual` → локальная запись + постановка в очередь репликации → немедленный возврат

## Рекомендуемый Порядок Реализации

### Фаза 1 — Транспортная абстракция

Ввести `LedgerNetworkBackend` как третий бэкенд рядом с `:memory` и `:file`.

```ruby
# Никаких изменений кода приложения — просто меняем бэкенд:
store = Igniter::Companion::Store.new(
  backend: :network,
  transport: :tcp,
  address:   "store.internal:7400"
)
```

`LedgerNetworkBackend` сериализует вызовы `write_fact` / `replay` / `write_snapshot`
через транспорт. Серверная сторона — минимальный процесс store-server.

Это **минимальный жизнеспособный шаг**, который открывает все топологии без
изменения фасада `Companion::Store`.

### Фаза 2 — SubscriptionRegistry

Перенести подписки `on_scope` из in-process Ruby Proc в постоянные объекты
`SubscriptionRecord`, хранящиеся на сервере.

```ruby
SubscriptionRecord = Struct.new(:store, :scope, :adapter, :config)
```

Event bus сервера читает `SubscriptionRegistry` после каждой записи и запускает
адаптеры доставки.

### Фаза 3 — Replication log

Добавить append-only `ReplicationLog` рядом с WAL. Каждая запись факта также
постится в replication log. Фоновый `ReplicationWorker` читает лог и рассылает
на реплики. Реплики применяют факты через `FactLog#replay` (без записи в WAL —
WAL принадлежит primary).

### Фаза 4 — Аннотация консистентности

Проводить метаданные `consistency:` через путь записи сетевого бэкенда.
`:strong` записи блокируются до кворумного подтверждения; `:eventual` возвращают
управление после локального коммита.

## Что Остаётся в Ruby / Приложении

Даже при полном кластерном деплое:

| Забота | Местонахождение |
|--------|----------------|
| Вычисление узлов контракта | Процесс приложения |
| DSL (input/compute/output) | Процесс приложения |
| Определение схем scope/history | Процесс приложения |
| Подключение адаптеров (`on_scope`) | Процесс приложения (DSL) |
| Хранение фактов, WAL, snapshot | Сервер хранилища |
| Индекс scope | Сервер хранилища |
| Event bus + доставка | Сервер хранилища |
| Репликация | Сервер → реплики |

Приложение всегда **писатель и читатель**; оно никогда не становится пассивным клиентом.
Сервер хранилища всегда **долговечный хост проекции**; он никогда не выполняет логику.

## Открытые Вопросы

1. **Формат протокола**: JSON-over-TCP vs MessagePack vs gRPC.
   Текущий WAL использует CRC32-framed MessagePack — повторное использование
   для транспорта естественно.

2. **Авторизация**: mTLS для server-to-server; API key или токен для app-to-server.
   Вне области Фазы 1 (сначала только localhost-транспорт).

3. **Обратное давление**: что происходит, когда replication log растёт быстрее,
   чем реплики потребляют? Нужна bounded queue или сигнал flow control.

4. **Репликация снапшотов**: реплики должны бутстрапиться из снапшота, не
   реплеить весь WAL. `write_snapshot` уже существует — повторное использование
   для бутстрапа реплики.

5. **Registry схем**: при нескольких инстансах приложения, как coercion-хуки и
   версии схем остаются консистентными? Долгосрочно: сервер держит schema registry;
   приложения регистрируются при подключении.

## Связь с Существующим Кодом

- `FileBackend` — уже имеет CRC32-framed WAL + snapshot. `LedgerNetworkBackend` переиспользует
  формат фреймирования для своего wire protocol.
- `FactLog#all_facts` — используется `checkpoint`; также нужен для бутстрапа
  снапшота при репликации.
- `CoercedFact` / `register_coercion` — в Фазах 2+, coercions могут мигрировать
  в server-side schema registry.
- Playground демо 05 (snapshot) и 06 (concurrency) — прямые предшественники
  поведения репликации и кластера.
